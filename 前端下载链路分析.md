# 前端能否直接用视频下载链接？

> 结论先行：**不能**。前端既拿不到真实链接，拿到也无法直接下载。
> 现在的「后端中转下载」是必要设计，本文说明原因并给出可落地的优化方案。

---

## 一、问题

希望前端拿到抖音视频的**真实下载直链**，直接通过该链接下载，省掉后端中转 / 异步任务那一套。

---

## 二、现状：代码现在是怎么走的

### 2.1 解析接口丢弃了链接

`/parse/video` 返回的 `qualities` 数组里**只有清晰度元数据，没有 `url`**。
见 [services/download_service.py:50-79](services/download_service.py#L50-L79)，构造 payload 时主动没放 `url`：

```python
payload = {
    "ratio": item.get("ratio"),
    "bit_rate": item.get("bit_rate", 0),
    "quality_label": ...,
    "gear_name": item.get("gear_name", ""),
}   # ← 没有 url
```

而后端自己下载时取的 `url` 来自**原始 info**（前端看不到的那层），见 [services/download_service.py:21-32](services/download_service.py#L21-L32)：

```python
return best.get("url")      # choose_download_url
return qualities[0].get("url")
```

### 2.2 前端只靠 ratio 选清晰度，下载全走后端中转

- 前端解析后只用 `ratio` 选择清晰度：[web/index.html:882-888](web/index.html#L882-L888)
- 下载走异步任务：创建任务 → 轮询 → 拉文件流
  - [web/index.html:1021](web/index.html#L1021)（创建任务 `POST /download/video/task`）
  - [web/index.html:973](web/index.html#L973)（轮询任务 `GET /download/video/task/{id}`）
  - [web/index.html:988](web/index.html#L988)（拉文件 `GET /download/video/task/{id}/file`）

> 也就是说：**前端目前完全不接触视频直链**，全部经由本地后端中转。

---

## 三、为什么不能直接把链接给前端

存在**两个独立的硬限制**，任意一个都足以让「前端直链下载」失败。

### 限制 1：后端没有把链接暴露给前端

见第二节 2.1，`serialize_video_info` 主动丢弃了 `url`。前端解析结果里根本不存在可下载地址。

### 限制 2：即使把链接硬塞给前端，浏览器也下不动

抖音视频 CDN 有 **Referer 防盗链**。后端下载时被强制要求带这两个头，见 [services/download_service.py:150-160](services/download_service.py#L150-L160)：

```python
headers = {
    "Referer": "https://www.douyin.com/",
    "Origin": "https://www.douyin.com",
    "User-Agent": "... Chrome ...",
    "Range": "bytes=0-",
}
```

不带就会被 CDN 拒绝（403）。而浏览器从前端页面（`localhost`）发起请求时：

| 方式 | 结果 |
|------|------|
| `<a href="直链" download>` / `window.open` | 浏览器自带 `Referer` 是 `localhost`，不是 `douyin.com` → **403** |
| `fetch(直链)` | 抖音 CDN 不返回 CORS 头 → **被浏览器跨域拦截**；且 Referer 问题依旧 |
| 任何 JS 手段伪造 `Referer` | 浏览器**禁止**改 `Referer`/`Origin`，无法绕过 |

三种方式全部失败。此外直链带签名**有时效**，过期即失效，前端缓存也无意义。

> 本质：**防盗链 + 跨域 CORS** 决定了浏览器不可能直连抖音 CDN 拉视频。只有「服务端带 header 拉取」这一条路。

---

## 四、解决方案

根据真实诉求分三种情况处理。

### 方案 A（推荐）：后端流式代理接口 + 前端 `<a>` 触发

**适用场景**：嫌异步任务太重，想要「点一下就下载」的直链体验。

**思路**：后端提供一个 **GET 流式代理接口**，前端用 `<a href>` 触发，浏览器原生下载（自带进度条、不占 JS 内存）。流量仍走本地后端，但本地服务（localhost）无带宽压力。

**关键改动点**：

1. 后端新增 GET 接口（流式返回，避免整文件进内存）—— 在 [api_server.py](api_server.py) 增加：

   ```python
   from fastapi.responses import StreamingResponse

   @app.get("/download/video/direct")
   def download_video_direct(url: str, quality: str | None = None, session_id: str | None = None):
       cookie = login_manager.get_cookie(session_id)
       if not cookie:
           raise HTTPException(status_code=401, detail="请先扫码登录")
       result = download_video(url, cookie=cookie, quality=quality)
       def iterfile():
           with open(result.path, "rb") as f:
               yield from f
       return StreamingResponse(iterfile(), media_type=result.content_type,
                                headers={"Content-Disposition": f'attachment; filename="{result.filename}"'})
   ```

2. 前端把「下载」按钮改成 `<a>` 触发（[web/index.html](web/index.html) 下载相关逻辑处）：

   ```html
   <a href="/download/video/direct?url=encodeURIComponent(链接)&quality=1080p&session_id=xxx">下载</a>
   ```

**注意**：
- `session_id` 走 query 参数有泄露风险，本地服务可接受；若介意，可不传，后端默认取已登录 cookie（[api_server.py:90-105](api_server.py#L90-L105) 已有 `get_cookie(None)` 的默认取法）。
- 这是「体验上的直链」，技术上仍是后端代理，**不是**抖音 CDN 直链。

### 方案 B（现状，保持）：异步任务中转

**适用场景**：需要前端自定义进度条、并发任务管理。

当前 [api_server.py:108-142](api_server.py#L108-L142) + [services/download_tasks.py](services/download_tasks.py) 已实现，**无需改动**。最稳，但流程最长。

### 方案 C：不走后端流量 —— 不可行

**结论**：Web 前端做不到。原因见第三节「限制 2」（防盗链 + CORS）。
若一定要绕过后端，只能脱离浏览器（如生成带 header 的 `curl`/`yt-dlp` 命令让用户在终端执行），这已超出 Web 前端能力范围。

---

## 五、结论

| 问题 | 答案 |
|------|------|
| 能把抖音视频直链直接给前端吗？ | **不能** |
| 原因 1 | 后端 `serialize_video_info` 主动丢弃了 `url`，前端解析结果里没有链接 |
| 原因 2 | 抖音 CDN 防盗链（强制 `Referer: douyin.com`）+ 跨域 CORS，浏览器无法直连 |
| 现状设计是否合理 | **合理且必要**，后端中转是唯一可行路径 |
| 推荐优化 | 方案 A：后端加 GET 流式代理接口，前端用 `<a>` 触发，体验等同直链且省掉任务轮询 |
