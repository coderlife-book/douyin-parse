# 抖音解析、预览、下载完整方案

本文基于当前项目 `/Users/sometimes/Desktop/douyin-parse` 的真实实现整理，可作为另一个系统实现抖音解析、视频预览、视频下载时的参考。

当前方案的核心原则：

- 前端只传“抖音分享文本/分享链接”和用户选择的清晰度，不直接持有 CDN 视频直链。
- 每次解析、预览、下载都由后端现场解析，拿到新鲜直链后立即消费。
- 解析接口只返回元信息和清晰度列表，不把真实视频 URL 暴露给前端。
- 预览走后端 Range 流式代理，解决浏览器直连抖音 CDN 的 CORS、防盗链、Referer 问题。
- 下载走后端异步任务，先落本地文件，再通过本地文件接口给前端保存。

## 1. 总体链路

```text
前端输入抖音分享文本/链接
  -> POST /parse/video
  -> 后端使用 Cookie + a_bogus/X-Bogus 访问 aweme detail
  -> 提取视频元信息、封面、清晰度、码率
  -> 前端展示清晰度
  -> 前端设置 <video src="/preview/video?...">
  -> 后端重新解析并代理 CDN Range 响应
  -> 用户点击下载
  -> POST /download/video/task
  -> 后端后台线程重新解析、选择清晰度、下载到本地
  -> 前端轮询任务进度
  -> GET /download/video/task/{task_id}/file 取本地 mp4
```

关键文件：

- `api_server.py`：FastAPI API 入口。
- `douyin_video_parser.py`：抖音分享链接解析、aweme detail 请求、视频地址/清晰度提取。
- `services/download_service.py`：解析结果序列化、清晰度选择、预览流、下载落盘。
- `services/download_tasks.py`：下载任务、进度、后台线程。
- `services/douyin_login.py`：扫码登录、Cookie 保存和读取。
- `web/index.html`：前端解析、预览、下载按钮逻辑。

## 2. Cookie 和登录

抖音 Web 接口需要有效 Cookie。当前项目使用 Playwright 打开抖音页面，用户扫码登录后保存 Cookie 到本地 `config.json`。

相关接口：

```http
GET /health
POST /auth/session
GET /auth/session/{session_id}
POST /auth/logout
```

关键逻辑：

- `GET /health` 返回本地是否已有 Cookie。
- `POST /auth/session` 创建扫码会话，返回 `session_id` 和二维码。
- 前端轮询 `GET /auth/session/{session_id}`，直到 `status=logged_in`。
- Cookie 保存后，后续解析/预览/下载都通过 `login_manager.get_cookie(session_id)` 获取。

登录成功判断不是只看一个 cookie，而是检查：

- 存在 `sessionid` 且存在 `sid_tt` 或 `uid_tt`。
- 或者 `passport_auth_status=1`。

迁移建议：

- 另一个系统不要依赖浏览器普通登录态能直接被后端读取，后端必须有自己的 Cookie 来源。
- Cookie 过期时，解析失败一般表现为 aweme detail 为空、401、403 或返回结构缺字段。
- Cookie 保存时要保留完整 `name=value; name2=value2` header 格式。

## 3. 解析方案

入口是 `DouyinVideoParser.parse_video(share_url)`。

### 3.1 从分享文本提取 URL/视频 ID

抖音复制出来的内容通常不是纯 URL，而是类似：

```text
6.10 06/29 ... # 美食探店 https://v.douyin.com/xxx/ 复制此链接，打开 Dou音搜索...
```

解析步骤：

1. 先用正则从整段文本里提取 URL：
   - `https?://...`
   - `v.douyin.com/...`
   - `douyin.com/video/...`
   - `douyin.com/aweme/detail/...`
2. 如果是短链，后端请求短链并跟随重定向。
3. 从最终 URL 或 HTML 中提取：
   - `/video/{id}`
   - `/aweme/detail/{id}`
   - `/note/{id}`
   - `video_id=`
   - `aweme_id=`
   - `note_id=`
   - HTML 中的 `"aweme_id":"..."`

迁移建议：

- 输入参数必须允许“整段分享文本”，不要要求用户手工只粘 URL。
- URL 提取时不要被中文、emoji、空格、`#`、换行干扰。
- 短链必须服务端 follow redirect，不能只靠前端。

### 3.2 请求 aweme detail

当前项目请求：

```text
https://www.douyin.com/aweme/v1/web/aweme/detail/
```

主要参数包括：

- `device_platform=webapp`
- `aid=6383`
- `channel=channel_pc_web`
- `aweme_id={video_id}`
- 浏览器、系统、语言相关参数
- `a_bogus`
- 失败时再尝试 `X-Bogus`

请求头：

```http
User-Agent: Mozilla/5.0 ... Chrome/90/124 ...
Referer: https://www.douyin.com/video/{aweme_id}
Cookie: {saved_cookie}
Accept: application/json, text/plain, */*
```

如果是 `/note/` 内容，Referer 使用：

```text
https://www.douyin.com/note/{id}
```

迁移建议：

- a_bogus/X-Bogus 失败时要有降级尝试。
- Referer 要和内容类型匹配，视频用 `/video/{id}`，图文/笔记可尝试 `/note/{id}`。
- Cookie、User-Agent、Referer 三者要一致，不要一个像移动端一个像 PC 端。

## 4. 解析结果结构

当前视频解析结果内部包含：

```json
{
  "aweme_id": "...",
  "desc": "...",
  "create_time": 123,
  "author_nickname": "...",
  "author_sec_uid": "...",
  "cover_url": "...",
  "content_type": "video",
  "nwm_url": "真实视频地址",
  "qualities": [
    {
      "url": "真实视频地址",
      "ratio": "1080p",
      "bit_rate": 2536000,
      "quality_label": "1080p (2536Kbps)",
      "gear_name": "normal_1080_0"
    }
  ]
}
```

但是 `/parse/video` 返回给前端时会刻意去掉 `url`，只保留：

```json
{
  "aweme_id": "...",
  "desc": "...",
  "create_time": 123,
  "author_nickname": "...",
  "author_sec_uid": "...",
  "cover_url": "...",
  "content_type": "video",
  "qualities": [
    {
      "ratio": "1080p",
      "bit_rate": 2536000,
      "quality_label": "1080p (2536Kbps)",
      "gear_name": "normal_1080_0"
    }
  ]
}
```

这样做的原因：

- CDN 直链可能短期过期。
- CDN 直链可能和 Cookie、IP、UA、Referer 有绑定。
- 直链给前端再回传，容易出现编码截断、过期、一次性链接失效。
- 前端不需要真实 URL，只需要清晰度枚举。

## 5. 清晰度选择

当前从 `aweme_detail.video.bit_rate` 提取清晰度列表。

优先级：

1. 遍历 `video.bit_rate[*].play_addr.url_list`。
2. 从 `quality_type.name`、`gear_name` 或码率推断 `ratio`。
3. 将 `playwm` 替换为 `play`，尽量使用无水印地址。
4. 如果没有 `bit_rate`，回退到 `video.play_addr.url_list`。
5. 如果还没有，使用 `play_addr.uri` 构造常见 ratio 地址。
6. 去重。
7. 按 `bit_rate` 和 `ratio` 排序，默认最高质量在前。

下载/预览时调用 `choose_download_url(info, quality)`：

- 如果用户选择了 `quality`，从同 ratio 中选 `bit_rate` 最大的。
- 如果没选，使用 `qualities[0]`。
- 如果没有 qualities，回退 `nwm_url`。

迁移建议：

- 不要只取第一个 `url_list[0]` 当作唯一方案，要保留清晰度列表。
- 同一个 ratio 可能有多个码率，应选码率高的。
- 前端展示的清晰度不要等同于真实下载 URL，真实 URL 后端内部使用。

## 6. 预览方案

前端预览：

```js
const params = new URLSearchParams({ url });
if (ratio) params.set("quality", ratio);
if (sessionId) params.set("session_id", sessionId);
video.src = `/preview/video?${params.toString()}`;
video.load();
```

后端接口：

```http
GET /preview/video?url={share_text_or_url}&quality=1080p&session_id={session_id}
Range: bytes=0-1023
```

后端处理：

1. 从 session 或本地配置读取 Cookie。
2. 重新解析分享链接。
3. 按 quality 选择视频 URL。
4. 请求 CDN 时带防盗链请求头。
5. 把浏览器的 `Range` 透传给 CDN。
6. 透传关键响应头给浏览器。
7. 用 `StreamingResponse` 返回生成器。

CDN 请求头：

```http
User-Agent: Mozilla/5.0 ... Chrome/90.0.4430.212 ...
Referer: https://www.douyin.com/
Origin: https://www.douyin.com
Accept: */*
Accept-Encoding: identity
Range: {browser_range_or_bytes=0-}
```

本地响应头至少保留：

```http
Content-Length: {cdn_content_length}
Content-Range: bytes start-end/total
Accept-Ranges: bytes
Content-Type: video/mp4
```

重要坑：

- `StreamingResponse` 必须拿到生成器实例，不是内部函数本身。
  - 正确：`return stream(), status_code, headers`
  - 错误：`return stream, status_code, headers`
- 如果返回了 `206 + content-length`，实际 body 字节数必须等于 `content-length`。
- 如果上游返回压缩内容，会导致声明长度和实际字节不一致，所以预览请求使用 `Accept-Encoding: identity`。
- 不要把浏览器 Range 忽略掉后始终请求 `bytes=0-`，浏览器会频繁 seek，必须尊重 Range。
- 不要让浏览器直接请求抖音 CDN，通常会被 CORS 或防盗链拦截。

当前刚修过的关键错误就是：返回了 `stream` 函数而不是 `stream()`，导致响应头已发出，但 body 迭代失败，浏览器报：

```text
ERR_CONTENT_LENGTH_MISMATCH 206 (Partial Content)
```

## 7. 下载方案

当前提供两种下载接口。

### 7.1 同步下载接口

```http
POST /download/video
Content-Type: application/json

{
  "url": "抖音分享链接",
  "session_id": "...",
  "quality": "1080p"
}
```

后端：

1. 校验 Cookie。
2. 重新解析分享链接。
3. 选择清晰度 URL。
4. 下载到本地 `downloads/`。
5. 用 `FileResponse` 返回 mp4。

### 7.2 异步任务下载接口

这是当前前端主要使用的下载方式。

创建任务：

```http
POST /download/video/task
Content-Type: application/json

{
  "url": "抖音分享链接",
  "session_id": "...",
  "quality": "1080p"
}
```

返回：

```json
{
  "task_id": "...",
  "status": "queued",
  "message": "等待下载",
  "progress": 0,
  "downloaded": 0,
  "total": 0
}
```

轮询进度：

```http
GET /download/video/task/{task_id}
```

下载完成后取文件：

```http
GET /download/video/task/{task_id}/file
```

后端任务状态：

- `queued`
- `running`
- `done`
- `failed`

下载请求 CDN 的 header：

```http
User-Agent: Mozilla/5.0 ... Chrome/90.0.4430.212 ...
Referer: https://www.douyin.com/
Origin: https://www.douyin.com
Accept: */*
Range: bytes=0-
```

进度算法：

- `total = int(response.headers.get("content-length", 0))`
- 每写一个 chunk 累加 `downloaded`
- `progress = downloaded * 100 / total`
- 如果 total 未知，进度保持 0 或用“未知大小”

迁移建议：

- 大文件不要让前端 fetch CDN，也不要把整个文件读入内存。
- 最稳的下载链路是后端落盘，完成后本地文件接口返回。
- 下载时也要重新解析，不要复用前端解析阶段拿到的旧直链。
- 文件名要清理非法字符，避免 `/ \ : * ? " < > |` 破坏路径。

## 8. “没声音”的重点排查

当前项目优先使用 `play_addr` 和 `bit_rate[*].play_addr` 提取 URL，通常拿到的是可直接播放/下载的 mp4。

如果另一个系统出现“下载后没声音”，重点查这些点：

1. 是否选到了视频-only 的 DASH 轨道。
   - 有些平台/字段会把 video track 和 audio track 分开。
   - 只下载 video track 就会没声音。
   - 解决：同时提取 audio URL，用 ffmpeg mux。

2. 是否误用了错误字段。
   - 优先使用 `video.bit_rate[*].play_addr.url_list` 或 `video.play_addr.url_list`。
   - 谨慎使用只含画面的编码轨道、动态封面、live photo 的 video 字段。
   - 不要把 `cover`、`dynamic_cover`、`origin_cover` 当视频。

3. 是否使用了图集/实况图的 video URL。
   - 图集 live image 可能也有 `video` 字段，但那不是普通视频作品的主视频。
   - 当前 Web 页面只支持 `content_type=video`，图集直接拒绝下载。

4. 是否 CDN 返回了非 mp4 或错误页面。
   - 保存后用 `file xxx.mp4` 或 `ffprobe xxx.mp4` 检查。
   - 如果实际是 HTML/JSON 错误页，播放器可能表现异常。

5. 是否 Content-Length mismatch 导致文件不完整。
   - 不完整 mp4 也可能只有画面、无法 seek、无声音或直接不能播。
   - 下载后要校验实际字节数和响应头 `content-length`。

建议另一个系统增加下载后校验：

```bash
ffprobe -v error -show_streams output.mp4
```

至少确认：

- 有一个 `codec_type=video`
- 有一个 `codec_type=audio`
- 文件时长合理
- 文件大小和下载进度一致

如果确实是音视频分离，后端应该实现：

```bash
ffmpeg -i video.mp4 -i audio.m4a -c copy output.mp4
```

## 9. 常见错误和对应根因

### 9.1 解析失败

常见原因：

- 输入不是有效抖音分享文本/链接。
- 短链重定向失败。
- Cookie 过期。
- aweme detail 需要新的 a_bogus/X-Bogus 算法。
- Referer 不匹配，例如 note 用了 video Referer。
- 内容是图集，但接口只支持视频。

排查：

- 打印提取到的 URL 和 aweme_id。
- 打印 aweme detail HTTP 状态码和返回前 200 字符。
- 确认 Cookie header 非空。
- 同一个 aweme_id 尝试 video 和 note Referer。

### 9.2 预览失败

常见原因：

- 浏览器直连 CDN，被 CORS 或防盗链拦截。
- 后端没有透传 Range。
- 后端返回 `206` 但 Content-Length 和实际 body 不一致。
- `StreamingResponse` 返回了函数而不是生成器。
- 上游使用压缩传输，导致长度不一致。
- CDN 直链过期或一次性链接被重复消费。

排查：

```bash
curl -H "Range: bytes=0-1023" \
  -D headers.txt \
  -o range.bin \
  "http://127.0.0.1:8787/preview/video?url=...&quality=1080p"

cat headers.txt
wc -c range.bin
```

期望：

```text
HTTP/1.1 206 Partial Content
content-length: 1024
content-range: bytes 0-1023/总大小
```

并且：

```text
wc -c range.bin -> 1024
```

### 9.3 下载失败

常见原因：

- 后端请求 CDN 没带 `Referer: https://www.douyin.com/`。
- User-Agent 被识别为脚本。
- 直链过期。
- 前端把直链回传时 URL 编码丢失。
- 下载中断但任务仍标记成功。

当前方案规避方式：

- 下载时重新解析。
- 直链不出后端。
- 请求 CDN 带 Referer、Origin、User-Agent、Range。
- 后端流式写本地文件。
- 前端只轮询任务，不直接碰 CDN。

## 11. 实现时的硬性约束

- 不要把 CDN 直链返回给前端。
- 不要前端请求 CDN。
- 不要解析一次后长期保存直链给下载复用。
- 不要只支持纯 URL 输入，要支持整段分享文本。
- 不要只取一个默认 URL，要提取清晰度列表。
- 不要忽略浏览器 Range。
- 不要返回错误的 `Content-Length`。
- 不要把 `StreamingResponse` 的内部生成器函数漏写成函数对象。
- 下载后要校验文件是否完整；遇到“没声音”必须检查音频 track。
