# 视频下载方案：CDN 拒绝问题分析与修复

> **现象**：视频下不下来。
> **当前链路**：后端解析得到视频直链 → **把链接返回前端** → 前端点下载，请求回到后端 → **后端再转发请求到 CDN** → **CDN 拒绝（通常 403）**。
>
> 本文聚焦：**为什么 CDN 拒绝 + 怎么正确地把视频下下来**。

---

## 一、CDN 为什么拒绝（根因逐项排查）

CDN（尤其带防盗链的视频 CDN）会校验请求的若干要素，后端转发时只要一项不符就被拒。按可能性排序：

### 1. Referer / Origin 不符 —— 最常见

CDN 防盗链：要求 `Referer` 是**源站域名**。后端用 HTTP 客户端转发时：
- 默认不带 `Referer`，或带的是后端自己的域名 → **被拒**
- 必须手动伪装成源站浏览器的 `Referer`

### 2. User-Agent 不符

部分 CDN 校验 UA 是真实浏览器。后端默认 UA（如 `python-requests/2.x`、`Go-http-client`）会被识别并拒绝。

### 3. 链接已过期

视频直链几乎都带签名 + `expire` 时间戳。当前链路是「后端解析 → 给前端 → 前端再请求 → 后端转发」，中间有人为操作和网络往返耗时，等真正发起下载时链接可能**已过期** → CDN 拒绝。

### 4. 链接是「一次性」的（IP / UA 绑定）

有些链接绑定**首次请求**的客户端指纹。后端在**解析阶段**往往已经请求过一次（探测 / 校验 / 拿信息），链接被标记消费，第二次请求（转发下载）就被拒。

### 5. query 参数丢失或被改写

前端拿到链接回传给后端时，如果 URL 编码错误 / 截断 / 丢失了签名参数 → 校验失败。JWT/签名里的 `+ / = / &` 字符尤其容易出问题。

## 二、直接修复：后端转发必须带正确 header

后端代理下载时，**伪装成源站浏览器**：

```python
headers = {
    "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) "
                  "AppleWebKit/537.36 (KHTML, like Gecko) "
                  "Chrome/120.0.0.0 Safari/537.36",
    "Referer": "https://源站域名/",        # ← 关键，必须是视频来源站的域名
    "Origin":  "https://源站域名",
    "Accept":  "*/*",
    "Range":   "bytes=0-",                 # 支持流式 / 断点
}
resp = requests.get(url, headers=headers, stream=True, timeout=30)
```

> 例如抖音源站，`Referer` 必须是 `https://www.douyin.com/`。源站是哪个，看解析出来的直链域名归属。

---

## 三、架构修复：链接不要经前端绕一圈（关键）

当前「后端把链接给前端 → 前端回传 → 后端转发」这条链路**本身就有害**，即使 header 带对了也容易失败，因为：
- 绕路耗时 → 链接**过期**（根因 3）
- 解析时已请求过 → 链接**一次性失效**（根因 4）

**原则：CDN 直链是「后端内部的一次性资源」，不应该交给前端再回传。**

### 正确做法：下载时后端重新解析，链接全程不出后端

```
前端：POST /download  body: { url: 分享链接 或 video_id }
后端：  parse(分享链接)  → 拿到「新鲜」直链
        → 立即带正确 header 下载
        → 流式返回给前端
```

- 前端只传**分享链接 / 视频 ID**，**不传 CDN 直链**。
- 链接生成后**立刻消费**，不过期、不会被消费两次。
- 解析和下载合并到同一次后端调用里，避免中间任何间隔。

```python
def download(share_url):
    info = parser.parse_video(share_url)          # 现场解析，拿新鲜链接
    url  = choose_download_url(info)              # 选清晰度
    headers = { "Referer": "https://源站/", "User-Agent": UA, "Range": "bytes=0-" }
    resp = requests.get(url, headers=headers, stream=True, timeout=30)
    return StreamingResponse(stream_chunks(resp), media_type="video/mp4",
                             headers={"Content-Disposition": 'attachment; filename="x.mp4"'})
```

> 这正是本仓库抖音项目 [services/download_service.py](services/download_service.py) 的做法：`download_video` 内部先 `parse_video` 重新解析，再立刻 `_download_file`，链接全程在后端。

## 四、流式转发，别整文件进内存

后端从 CDN 拿到响应后，**边收边发**给前端，不要先下完整文件再返回（大视频会爆内存 / 超时）：

```python
def stream_chunks(resp):
    for chunk in resp.iter_content(chunk_size=64 * 1024):
        if chunk:
            yield chunk
```

## 五、当下排查清单（先定位是哪个根因）

1. **看 CDN 响应码**：
   - `403` → 多半是防盗链（Referer/UA 没带对）
   - `404 / 410 / 403 + media not found` → 多半是链接过期 / 一次性失效
2. **抓后端实际发出的请求 header**，确认 `Referer` / `User-Agent` 是否带上了源站值（不是后端自己的域名）。
3. **用 curl 直接验证链接本身**：
   ```bash
   curl -H "Referer: https://源站/" \
        -H "User-Agent: Mozilla/5.0 ... Chrome/120 ..." \
        -H "Range: bytes=0-" \
        "直链" -o test.mp4 -i
   ```
   - curl **能下**、后端下不了 → 后端 header 没带对（根因 1、2）
   - curl **也下不了** → 链接已过期 / 一次性（根因 3、4），需重新解析拿新链接
4. 对比「解析时的链接」和「下载时的链接」是不是同一条、间隔多久——若间隔超过几十秒还失败，基本是过期或一次性。

---

## 六、架构对比

| | 当前做法 | 正确做法 |
|---|---|---|
| 链接流向 | 后端 → 前端 → 后端 | 后端内部持有，不出后端 |
| 前端传什么 | CDN 直链 | 分享链接 / 视频 ID |
| 过期风险 | 高（绕路耗时） | 低（立即消费） |
| 一次性链接 | 易失效 | 不会 |
| Referer / UA | 常漏带 | 强制带源站值 |
| 内存 | 可能整文件入内存 | 流式转发 |

---

## 七、结论

- **直接原因**：后端转发请求没满足 CDN 防盗链要求（`Referer` / `User-Agent`），或链接已过期 / 一次性失效。
- **两个必须做的修复**：
  1. 后端转发带**源站 `Referer` + 真实浏览器 `User-Agent`**；
  2. **链接别经前端绕路**——下载时后端重新解析、立即消费，前端只传分享链接 / 视频 ID。
- **先按第五节排查清单**用 curl 定位是 header 问题还是链接失效，再对症改。

> 附：如果下载接口本身还要 token 鉴权（前端 → 后端这一段），那是普通接口鉴权，前端按 header 正常传给后端即可，跟 CDN 拒绝无关，不在此文展开。
